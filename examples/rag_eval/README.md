# RAG vs no-RAG — оценка качества

Агент jarvis в двух режимах (с RAG / без RAG) + контрольные наборы вопросов
и автоматическое сравнение качества.

## Как устроено

`evaluate.py` на каждый вопрос дважды дёргает LLM-клиент jarvis:

- **no-RAG** — вопрос уходит в модель как есть (пустой системный промпт);
- **RAG** — к вопросу через `build_system_prompt(retrieval_engine=...)`
  подмешиваются top-k фрагментов, найденных в FAISS-индексе (эмбеддинги
  bge-m3 через Ollama).

Метрики:

| Метрика | Что меряет |
|---|---|
| **hit@k** | попал ли ожидаемый источник в найденные чанки (только RAG) |
| **keywords** | доля ожидаемых ключевых слов, встретившихся в ответе |
| **judge** | оценка LLM-судьи 0–2 ответа против эталонного «ожидания» |

Наборы вопросов (по 10, с полями `question`, `expectation`,
`expected_keywords`, `expected_sources`):

- `questions.json` — по публичной документации Kotlin;
- `questions_jarvis.json` — по приватному коду самого jarvis-cli.

## Запуск

```bash
# из корня jarvis, в venv с extra [rag]; требуется Ollama + ключ провайдера в .env
./.venv/bin/python examples/rag_eval/evaluate.py                       # корпус Kotlin
./.venv/bin/python examples/rag_eval/evaluate.py \
    --index-path ~/rag-kotlin/index_jarvis --strategy structural \
    --questions questions_jarvis.json                                  # корпус jarvis
```

## Результаты и главный вывод

**Корпус 1 — публичные доки Kotlin:**

| | no-RAG | RAG |
|---|---|---|
| hit@5 | — | 100% |
| keywords | 92% | 90% |
| judge (0–2) | **2.00** | **2.00** |

**Корпус 2 — приватный код jarvis-cli:**

| | no-RAG | RAG |
|---|---|---|
| hit@5 | — | 100% |
| keywords | 10% | 97% |
| judge (0–2) | **0.20** | **2.00** |

Вывод: **RAG повышает качество ровно там, где у модели нет знаний.**
Публичный Kotlin модель (DeepSeek) знает наизусть — RAG не добавляет ничего
(2.0 vs 2.0). Приватный код jarvis модель не видела — без RAG она
галлюцинирует (0.2/2.0), а с RAG отвечает точно и со ссылками на файлы
(2.0/2.0). Это фундаментальное свойство RAG: ценность = закрытие пробела
в знаниях модели, а не «улучшение» и без того известного.

## Кросс-язычность

Вопросы русские, корпус английский (доки Kotlin) — поэтому эмбеддинг-модель
должна быть мультиязычной. С `nomic-embed-text` русский запрос про «вторичный
конструктор» промахивался мимо `classes.md`; замена на **bge-m3** (мультиязычная,
1024-dim) дала hit@5 = 100%.

# Улучшенный RAG: фильтр + реранкинг + query rewrite

Второй этап: между поиском и LLM встроен многоступенчатый конвейер
(`app/retrieval_pipeline.py`, реализует тот же порт `RetrievalEngine`):

```
query → rewrite → поиск (fetch_k) → порог min_score → реранк → top_k
```

- **query rewrite** (`infra/query_rewriter.py`) — LLM переформулирует вопрос в
  развёрнутый поисковый запрос (термины, синонимы, раскрытые аббревиатуры);
- **порог min_score** — отсекает чанки с косинусом ниже порога;
- **реранк** (`infra/rerankers.py`) — `none` | `heuristic` (лексика + MMR, без
  модели) | `llm` (cross-encoder: модель оценивает релевантность каждого чанка).

Всё настраивается на лету через `/rag` (reranker · rewrite · threshold ·
fetchk · topk) или из `.env` (`RAG_*`).

## Метрика

Качество ответа (keywords/judge) шумит из-за недетерминизма генерации, поэтому
реранк оценивается **детерминированной retrieval-метрикой**: `MRR` — средний
обратный ранг первого ожидаемого источника в выдаче (1/ранг), и `hit@k`. Она
изолирует эффект реранка от шума LLM. `--no-judge` выключает генерацию ответов
вовсе — прогон становится быстрым, бесплатным и воспроизводимым.

## Запуск

```bash
# сравнение ступеней baseline → +filter → +rerank → improved (детерминированно)
./.venv/bin/python examples/rag_eval/evaluate.py --compare-modes --no-judge \
    --index-path ~/rag-kotlin/index_jarvis_full --strategy structural \
    --questions questions_jarvis_hard.json \
    --reranker heuristic --fetch-k 20 --min-score 0.45
# то же с LLM-реранкером и с судьёй (end-to-end, дороже и шумнее)
./.venv/bin/python examples/rag_eval/evaluate.py --compare-modes \
    --questions questions_jarvis_hard.json --reranker llm --min-score 0.45
```

## Результаты (корпус jarvis, 10 сложных cross-file вопросов, top_k=5)

| режим | hit@5 | MRR | комментарий |
|---|---|---|---|
| baseline | 100% | **0.82** | сильный bi-encoder bge-m3 — retrieval почти насыщен |
| +filter (0.45) | 100% | 0.82 | на on-topic **no-op** (все score выше порога) |
| +rerank(heuristic) | 80% | 0.68 | **вредит**: лексика демотит семантически верные чанки |
| +rerank(llm) | ~100% | ~0.82 | нейтрален (есть прогон-к-прогону разброс) |
| improved (llm + rewrite) | 100% | **0.85** | лучший, но выигрыш маргинальный |

**Где ступени реально работают:**

- **Порог — для off-topic, а не для реранка on-topic.** On-topic релевантные
  чанки дают косинус **0.51–0.73**, вопросы вне корпуса («борщ», «погода»,
  «футбол») — **0.35–0.44**. Есть разделяющий зазор ~0.45. С порогом 0.45
  off-topic-запрос отдаёт **0 чанков** (пустой контекст → модель честно говорит
  «нет в базе», без галлюцинаций), а без фильтра — 5 мусорных чанков. На on-topic
  порог 0.45 ничего не портит. Порог **надо калибровать под корпус/модель**:
  0.40 был слишком низким (не резал off-topic), выше 0.5 — начинает терять нужное.
- **Реранк помогает, когда есть запас.** При top_k=5 нужный файл и так в выдаче
  (hit 100%) — реранжировать нечего. При **top_k=3** baseline промахивается
  (hit 90%: h06 стоит на #5 по косинусу), а `improved` (rewrite + LLM-реранк)
  вытягивает его в топ-3 → **hit 100%, MRR 0.83**.
- **Эвристика ≠ бесплатный выигрыш.** Лексический реранк хорош для точных
  идентификаторов, но на «концептуальных» вопросах (где нужный файл использует
  другую лексику, чем вопрос) демотит правильные чанки. Для семантического
  корпуса лучше `none` или `llm`.
- **Query rewrite обоюдоостр.** Он меняет то, что находится: усиливает
  направление реранкера (с эвристикой топит сильнее: 0.68→0.50; с LLM при
  top_k=3 вытягивает промах: 90%→100%).

**Главный вывод второго этапа:** ценность фильтра/реранка/rewrite **зависит от
того, есть ли у baseline запас для улучшения**. На сильном bi-encoder с
on-topic-корпусом retrieval уже near-ceiling — второй этап даёт мало и может
навредить (особенно дешёвая эвристика). Реальные выигрыши: фильтр защищает от
off-topic-галлюцинаций, а LLM-реранк + rewrite окупаются при тесном top_k или
слабом ретривере — ценой дополнительных вызовов модели. Универсального «включи и
станет лучше» нет: ступени надо подбирать под корпус и мерить.

Наборы вопросов для этого этапа: `questions_jarvis_hard.json` (10 сложных
cross-file), `questions_offtopic.json` (3 вне корпуса — для демонстрации порога).

# Обязательные источники + цитаты + режим «не знаю»

Третий этап: ответ модели обязан содержать **три части** — сам ответ,
**источники** (`source · раздел (chunk_id)`) и **цитаты** (дословные фрагменты из
найденных чанков). Плюс усиление: если релевантного контекста нет (порог отсёк
всё), ассистент обязан сказать **«не знаю»** и попросить уточнение — вместо того
чтобы отвечать из общих знаний и галлюцинировать.

## Как это устроено

- **Формат ответа** задаёт `app/system_prompt.py::_RAG_HEADER`: жёсткий шаблон
  «ответ со ссылками `[i]` → Источники → Цитаты». Каждому найденному фрагменту в
  промпт кладётся заголовок `[i] source · раздел (chunk_id)`, чтобы модели было
  что процитировать точь-в-точь и на что сослаться.
- **Режим «не знаю»** — чистая логика в `build_system_prompt`: если RAG включён,
  но поиск с порогом `min_score` вернул **0 чанков**, вместо RAG-блока
  подставляется блок-инструкция `_RAG_WEAK_HEADER` («контекста нет → ответь
  "Не знаю", не выдумывай, источников не приводи»). Порог — тот же `min_score`
  (ручка `/rag threshold`), что калибровали на втором этапе (0.45).
- **Проверка** — `app/answer_check.py` (чистые функции, без LLM): парсит секции
  «Источники»/«Цитаты» (терпит markdown-обрамление `**…**`, `### …`), достаёт
  цитаты из кавычек и сверяет каждую с текстом чанков:
  - **дословно (grounded)** — цитата (или каждый её фрагмент вокруг `…`) буквально
    есть в чанке;
  - **из контекста (no_fabrication)** — если не дословно, но ≥70% значащих токенов
    цитаты встречаются в контексте → это реконструкция реального контента, а не
    выдумка; ниже порога → помечается как **fabricated**.

## Запуск

```bash
# 10 сложных on-topic вопросов: источники + цитаты + обоснованность
./.venv/bin/python examples/rag_eval/evaluate.py --check-citations \
    --index-path ~/rag-kotlin/index_jarvis_full --strategy structural \
    --questions questions_jarvis_hard.json --min-score 0.45

# режим «не знаю» на вопросах вне корпуса
./.venv/bin/python examples/rag_eval/evaluate.py --check-citations \
    --index-path ~/rag-kotlin/index_jarvis_full --strategy structural \
    --questions questions_offtopic.json --min-score 0.45 --show-answers
```

## Результаты (deepseek-chat, порог 0.45)

| проверка | on-topic (10) |
|---|---|
| источники в каждом ответе | **10/10 (100%)** |
| цитаты в каждом ответе | **10/10 (100%)** |
| цитаты **дословны** | 3–4/10 (~30–40%) |
| **без выдумок** (контент из контекста) | **10/10 (100%)** |

| проверка | off-topic (3) |
|---|---|
| честный отказ «не знаю» | **3/3 (100%)** |
| без выдуманных источников | **3/3 (100%)** |

## Выводы

- **Обязательный формат работает: 100% ответов несут источники и цитаты.** Раньше
  header лишь «просил» ссылаться — модель делала это непоследовательно; жёсткий
  трёхчастный шаблон + метаданные (`chunk_id`) в каждом фрагменте дают 10/10.
- **Выдумок нет, но дословность — только ~⅓.** Каждая цитата по содержанию взята
  из контекста (no_fabrication 100%), однако модель склонна **реконструировать**
  фрагмент (перефраз, КАПС, склейка UI-строк и списков), а не копировать. Поэтому
  строгий verbatim-grounding честно показывает ~30–40%. Вывод: «цитаты» LLM без
  проверки доверять нельзя — их надо валидировать дословным сопоставлением с
  чанками (что и делает `answer_check`). Это и есть ответ на вопрос «совпадает ли
  смысл ответа с цитатами»: содержательно да (100%), буквально — не всегда.
- **Режим «не знаю» надёжен.** На вопросах вне корпуса порог 0.45 отдаёт пустой
  контекст → модель в 3/3 случаях честно отказывается и **не выдумывает
  источников**. Это прямое усиление против галлюцинаций: раньше при пустом
  результате RAG-блок просто не добавлялся и модель отвечала из общих знаний.
- **Замечание о детерминизме.** Число цитат и доля дословных слегка плавает
  прогон-к-прогону даже при `temperature=0` (DeepSeek не строго детерминирован),
  поэтому наличие источников/цитат меряем как жёсткий факт, а grounding — как
  диапазон.

Модуль проверки покрыт юнит-тестами (`tests/app/test_answer_check.py`):
markdown-заголовки, эллипсис-выдержки, выдумка vs реконструкция, детект «не знаю».
